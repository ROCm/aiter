# GEMM Operators

## Overview

aiter provides multiple GEMM (General Matrix Multiply) operator families, each targeting a specific precision/quantization format. The general formula is D = alpha * A * B + C. All GEMM ops use the `@compile_ops` decorator to JIT-compile HIP/CK C++ extensions, with tuned configurations loaded from CSV files in `aiter/configs/`.

## GEMM Families

### A8W8 (FP8 activations x FP8 weights)

- Module: `gemm_op_a8w8.py`
- Key functions:
  - `gemm_a8w8(XQ, WQ, x_scale, w_scale, Out, bias, splitK)` -- basic FP8 GEMM via CK
  - `gemm_a8w8_bpreshuffle_ck(...)` -- with B-matrix pre-shuffle optimization
  - `gemm_a8w8_blockscale_ck(...)` -- with block-level scaling
  - `gemm_a8w8_blockscale_bpreshuffle_ck(...)` -- combined block-scale + pre-shuffle
- Config files: `a8w8_tuned_gemm.csv`, `a8w8_blockscale_tuned_gemm.csv`, `a8w8_bpreshuffle_tuned_gemm.csv`, `asm_a8w8_gemm.csv`
- The `gemm_a8w8()` wrapper function selects between CK and ASM backends based on tuned configs, with automatic M-padding and splitK selection.

### A16W16 (BF16/FP16 activations x BF16/FP16 weights)

- Module: `gemm_op_a16w16.py`
- Key function: `gemm_a16w16_asm(A, B, out, bias, splitK, kernelName, bpreshuffle)` -- ASM backend
- Uses a semaphore workspace for splitK synchronization
- Config file: `bf16_tuned_gemm.csv`

### A4W4 (INT4/FP4 activations x INT4/FP4 weights)

- Module: `gemm_op_a4w4.py`
- Key functions: `gemm_a4w4_blockscale(...)`, with CK and ASM variants
- Uses `get_GEMM_config(M, N, K)` to look up tuned kernel name and splitK from `a4w4_blockscale_tuned_gemm.csv`
- Supports M-padding via `get_padded_m()` with granularity levels
- FlyDSL backend available as optional alternative (see `flydsl_gemm_a4w4()`)

### Batched GEMM

- `batched_gemm_op_a8w8.py` -- batched FP8 GEMM
- `batched_gemm_op_bf16.py` -- batched BF16 GEMM
- Config files: `a8w8_tuned_batched_gemm.csv`, `bf16_tuned_batched_gemm.csv`

### DeepGEMM

- Module: `deepgemm.py`
- `deepgemm(XQ, WQ, Y, group_layout, x_scale, w_scale)` -- CK-based grouped GEMM
- Used for grouped/expert GEMM operations (MoE context)

## Configuration System

Tuned configs are stored as CSV files in `aiter/configs/`. Each CSV contains:

- `cu_num` -- number of compute units (GPU-specific, e.g., 304 for MI300X)
- `M`, `N`, `K` -- matrix dimensions
- `kernelName` -- which kernel implementation to use
- `splitK` -- split-K parallelism factor

At runtime, `get_GEMM_config(M, N, K)` reads the CSV, indexes by (cu_num, M, N, K), and returns the best config. If no tuned config exists, a default is used. The CSV files are generated by the [[concepts/autotuning]] pipeline.

- `a8w8_tuned_gemm.csv` / `a8w8_untuned_gemm.csv`
- `a8w8_blockscale_tuned_gemm.csv` / `a8w8_blockscale_untuned_gemm.csv`
- `a8w8_bpreshuffle_tuned_gemm.csv` / `a8w8_bpreshuffle_untuned_gemm.csv`
- `a8w8_blockscale_bpreshuffle_tuned_gemm.csv` / untuned variant

## Backend Selection

GEMM ops typically choose between backends in this priority:

1. **ASM** -- hand-tuned assembly kernels, highest performance for specific shapes
2. **CK** -- Composable Kernel C++ templates, good general coverage
3. **CK Tile** -- newer CK tile-based kernels (block-scale variants)
4. **Triton** -- available for some GEMM variants, used in MoE context

The selection is driven by the tuned CSV configs: if the config specifies an ASM kernel name, ASM is used; otherwise CK.

## M-Padding

`get_padded_m(M, N, K, granularity_level)` rounds up M to improve GEMM tile efficiency. This is critical for small-M (decode) scenarios where M doesn't align with tile sizes. The padding level is automatically selected when looking up configs.
